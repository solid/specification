Abstract

This document is the Solid26 Implementors Guide — a snapshot of the most mature and widely implemented Solid specification versions, collected to help developers and organisations identify a common baseline. It provides practical guidance for implementing the Solid specifications included in this collection.

Introduction

The Solid Technical Reports comprise multiple specification documents with differing levels of maturity. The Solid Protocol bundles the specifications that, together, provide applications with secure and permissioned access to externally stored data in an interoperable way. The Solid26 Implementors Guide points implementers to fixed versions of the Solid Protocol and its sub-specifications, a stable snapshot of the specifications to build against today. The Solid26 Implementors Guide selects for the most widely implemented specification versions at the time of this documents publication.

This document does not define new normative requirements, but rather identifies and collects the specifications needed to build interoperable Solid applications and servers.

Specifications

The Solid26 Implementors Guide recommends the following specification snapshots.

Solid Protocol

The Solid Protocol (CG-DRAFT, v0.11.0, 2024-05-12) is included with the following comments:

  • Clients might encounter Servers that do not support § 5.3.1 Modifying Resources Using N3 Patches despite it being required by the Solid Protocol. Clients can use Allow headers to determine if a Server supports HTTP PATCH requests.

    If a server does not support HTTP PATCH requests, the client can HTTP GET the resource, apply the change locally, and then HTTP PUT the resource back. If the server supports ETag headers or Last-Modified headers, clients can achieve the concurrency-control behaviour of HTTP PATCH by issuing a conditional PUT. ETag-based validation is more precise than date-based validation and should be preferred when a server supports both, because Last-Modified has only one-second resolution and may not reflect sub-second changes.

    The flow is:

    • On GET, the server returns an ETag — an opaque validator identifying the current state of the resource (often derived from a content hash, but the exact derivation is up to the server). For example, imagine the server returns ETag: "xyzzy".
    • The client then makes a conditional PUT using the If-Match header: If-Match: "xyzzy". The server applies the PUT only if the resource's current ETag still matches "xyzzy"; otherwise it responds with 412 Precondition Failed. This guarantees the resource has not changed on the server between the GET and the PUT.
    • The equivalent using dates is If-Unmodified-Since paired with the Last-Modified value from the GET response. Note that If-Modified-Since is the header used for cache revalidation on GET, not for lost-update prevention on PUT.

    Note that If-Match uses strong comparison, so weak ETags (those prefixed with W/) will not match.

  • Servers are strongly encouraged to implement Web Access Control (WAC), see below.

    Note

    The March 2026 implementation survey yields the following results (archived):

    • For WAC, the data shows 13 server-side implementations, deployment in 11 services, and 19 client-side implementations. WAC is pragmatic, user-friendly, and extensible. WAC has all the features that most open, social, Web applications need.
    • For ACP, the data shows 4 server-side implementations, deployment in 1 service, and 4 client-side implementations. ACP is highly expressive. ACP is most commonly applied in enterprise and B2B applications of Solid involving complex permissions.

    The data shows that most clients implement only one access control language, despite the Solid Protocol requiring Clients to conform to both WAC and ACP.

  • While the Solid Protocol technically requires any Client to conform to both authorization languages, Client implementers are advised to consider whether their Client implementation should actually attempt to modify access rules at all: A separation between a client executing application-specific logic and a client executing authorization-related logic might be beneficial in terms of separation of concerns, maintainability, and re-usability of software components [BKY+24]. Such approach might rely on
    • access requests to update access control rules accordingly on a Server
    • issued by the application-logic client
    • processed by either a particular Client that is able and trusted to manage access controls (such as an access management or authorization application) or by custom Server functionality

    Access requests and their processing are currently are poposed work item to the CG. Different approaches exist within the community; no consensus has been reached yet. Implementers are encouraged to provide their implementation expierence as input to the community's discussion.

Implementation Guidance

WebID

This section gives implementation guidance for WebID in the context of the Solid26 Implementors Guide. It is organised in three parts: a summary of the requirements that can be derived directly from the specifications pinned by the Solid26 Implementors Guide (§ 3.1.1); a summary of properties that are commonly assumed but are not strictly required by those specifications (§ 3.1.2); and concrete guidance for implementers (§ 3.1.3).

For the purposes of this section, the following terms are used:

WebID
An IRI that denotes an agent [WEBID], [SOLID-PROTOCOL].
WebID Profile
An RDF graph [RDF11-CONCEPTS] that contains information about the agent denoted by a WebID.
WebID Document
A Web resource that, when dereferenced, returns a representation of a WebID Profile.

Requirements derivable from Solid26 Implementors Guide specifications

The following requirements are directly stated by, or are direct logical consequences of, the specifications pinned by the Solid26 Implementors Guide (see § 2 Specifications).

  1. A WebID is an HTTP URI denoting an agent (e.g. a person, organisation, or piece of software) [WebID 1.0 § 2 Terminology, normative; Solid Protocol § 9.1 WebID, non-normative; Solid-OIDC § 3.1 WebIDs, non-normative].
  2. When a WebID is dereferenced, the server provides a representation of a WebID Profile as an RDF document [Solid-OIDC § 6 WebID Profile, normative; Solid Protocol § 9.1 WebID, non-normative].
  3. The WebID Profile MUST contain one or more statements of the form ?webid solid:oidcIssuer ?iss, where ?webid is the WebID and each ?iss is the IRI of an OpenID Provider trusted to issue ID Tokens for that WebID [Solid-OIDC § 6.1 OIDC Issuer Discovery, normative].
  4. The webid claim in a Solid-OIDC DPoP-bound ID Token MUST be the user's WebID. Verifiers MUST check that the token's iss claim matches one of the WebID's discovered solid:oidcIssuer values; otherwise, the OpenID Provider is not authoritative for that WebID [Solid-OIDC § 8.1 DPoP-bound OIDC ID Token, normative; Solid-OIDC § 6.1 OIDC Issuer Discovery, normative].
  5. A Solid-OIDC OpenID Provider declares conformance by advertising webid in the scopes_supported property of its OpenID Connect Discovery 1.0 metadata. Clients can therefore determine whether a discovered issuer is a Solid-OIDC OpenID Provider by retrieving its provider configuration document and checking for that value [Solid-OIDC § 10 Solid-OIDC Conformance Discovery, normative].
  6. A WAC server SHOULD NOT trust assertions found by dereferencing an agent's WebID for access-control decisions; only the WebID itself (and information the server independently trusts) is reliable for authorisation [Web Access Control § 8.1 Security Considerations, non-normative].
Note

The Solid Protocol pinned by the Solid26 Implementors Guide does not require a WebID Profile to contain a pim:storage statement pointing at the agent's storage. The protocol's storage-discovery mechanism ([Solid Protocol § 4.1 Storage Resource, normative]) describes how a client may discover a storage from any RDF representation containing a pim:storage relation, and from the Link header with rel="http://www.w3.org/ns/solid/terms#storageDescription" on storage resources. It does not bind storage discovery to the WebID Profile specifically.

Common assumptions not required by Solid26 Implementors Guide specifications

The following properties are widely assumed by existing Solid software and are encouraged by the underlying [WebID 1.0] specification, but are not stated as normative requirements by any specification pinned by the Solid26 Implementors Guide.

  1. The WebID Document is publicly readable (i.e. an unauthenticated HTTP GET on the WebID URI returns the Profile) [WebID 1.0 § 6 Processing the WebID Profile, non-normative].
  2. The WebID Document is available as text/turtle. Solid-OIDC and the Solid Protocol require only that the Profile be expressed in some RDF 1.1 serialisation; Turtle is the format described in WebID 1.0 processing rules and is the de-facto default in the existing ecosystem [WebID 1.0 § 6 Processing the WebID Profile, non-normative; Solid-OIDC § 3.1 WebIDs, non-normative].
  3. Dereferencing the WebID URI yields an HTTP 200 or a 303 redirect (per the "Cool URIs for the Semantic Web" pattern referenced by WebID 1.0) [WebID 1.0 § 3 The WebID HTTP URI, non-normative; Cool URIs for the Semantic Web § 4, non-normative].
  4. The WebID Document is served by a Solid storage. WebID Profiles can be hosted on any HTTP resource that can serve RDF; nothing in the Solid26 Implementors Guide requires the WebID Document to itself be a Solid resource.
  5. The WebID Profile contains a pim:storage statement linking to the agent's Solid storage. This is a common convention in existing servers but is not required (see note in § 3.1.1).
  6. The WebID URI uses a fragment identifier (a "hash URI"). Both fragment and non-fragment ("303") WebIDs are permitted by the underlying specifications.
  7. The WebID Document supports CORS so that browser-based clients can fetch it cross-origin. CORS support is not mandated by any specification pinned by the Solid26 Implementors Guide for arbitrary WebID hosts, although the Solid Protocol mandates CORS for resources served by Solid storages.
  8. A WebID Profile contains exactly one solid:oidcIssuer statement and at most one pim:storage statement. Solid-OIDC explicitly permits "one or more" issuers, and neither cardinality is otherwise constrained [Solid-OIDC § 6.1 OIDC Issuer Discovery, normative].

Implementation guidance

The following guidance is offered to implementers of clients and servers that consume WebIDs.

Treat WebIDs as plain HTTP resources

Because Solid26 does not require WebID Documents to be Solid resources, clients should dereference WebID URIs as plain HTTP resources. In particular, clients should not send an Authorization header or DPoP proof when fetching a WebID Document, since some hosts may reject authenticated requests for resources that do not expect them.

Content negotiation

Clients should send an Accept header advertising the RDF serialisations they can parse. text/turtle is the most widely deployed serialisation and is recommended as the primary value; clients that also support application/ld+json, application/n-triples, or application/rdf+xml should include those values too. Clients should be prepared to receive any RDF 1.1 serialisation in response and should fail gracefully if the returned representation cannot be parsed.

HTTP response handling

Clients should be prepared to handle, in addition to a direct 200 OK response, an HTTP 303 See Other redirect to the WebID Document, as described by the "Cool URIs for the Semantic Web" pattern. Other redirect status codes (e.g. 301, 302, 307, 308) may also be encountered and should be followed using standard HTTP semantics. A 401 response indicates the WebID Document is not publicly readable; in that case the WebID cannot be used for unauthenticated discovery and the client should surface a clear error to the user.

Fragment identifiers

Clients should distinguish between (a) the WebID itself, which includes any fragment identifier, and (b) the URI of the WebID Document, which is the WebID with the fragment removed (or the URI returned by following a 303 redirect). Authentication and identification always refer to the WebID, not the Document URI. When querying the Profile graph, the subject of statements about the agent is the full WebID (with fragment), not the Document URI.

Discovering the OpenID Provider

To discover the trusted OpenID Provider(s) for a WebID, clients dereference the WebID and evaluate the SPARQL pattern ?webid <http://www.w3.org/ns/solid/terms#oidcIssuer> ?iss over the resulting graph. If the Profile lists more than one issuer, the client should let the user select which to use. solid:oidcIssuer values transmitted in HTTP Link headers may be used as an optimisation but the RDF body remains canonical; clients that use Link headers should reconcile any discrepancy by relying on the body.

Clients should verify that a discovered issuer is a Solid-OIDC OpenID Provider by retrieving its OpenID Connect Discovery 1.0 configuration document and checking that webid appears in scopes_supported (Solid-OIDC § 10). If the issuer is unreachable or does not advertise Solid-OIDC conformance, the client should report this to the user with a meaningful error.

ID Token validation

When verifying a Solid-OIDC ID Token presented by a client, resource servers MUST check that the token's iss claim matches a solid:oidcIssuer value discoverable from the WebID identified by the token's webid claim. A token whose issuer is not authoritative for the asserted WebID MUST be rejected.

Discovering storage

Although a pim:storage statement in the WebID Profile is the most common way for a client to find an agent's Solid storage, it is not required. When the Profile does not contain such a statement, clients can fall back on the storage-discovery mechanisms defined by the Solid Protocol: requesting an arbitrary resource that may belong to a storage and reading the Link header with rel="http://www.w3.org/ns/solid/terms#storageDescription", or walking up the URI hierarchy looking for a resource typed as pim:Storage. A WebID Profile may list multiple storages; clients should be prepared to present a choice.

Extended profiles

A WebID Profile may reference additional documents via rdfs:seeAlso. When such references exist, clients may dereference them and treat their contents as part of the agent's Profile. Clients should be prepared for the case in which extended profiles are absent, unreachable, or contain statements that conflict with the primary Profile; per WAC guidance, statements from such documents should not be relied on for access-control decisions.

Validating user-supplied WebIDs

When asking users to enter a WebID, clients should validate that the input is a syntactically well-formed HTTP or HTTPS URI before attempting to dereference it. Clients should provide clear feedback for malformed input, for URIs that return 404, and for URIs that return responses that cannot be parsed as an RDF graph.

Security considerations for dereferencing

Dereferencing a WebID requires an application to make an HTTP request to a URI supplied by an untrusted party, and to parse the resulting content. Implementers should treat this as a server-side request forgery (SSRF) and untrusted-input surface and apply mitigations appropriate to the deployment, including (but not limited to) restricting the URI scheme to http/https, blocking requests to private and link-local IP ranges where appropriate, applying timeouts and response-size limits, bounding parser memory and CPU use, and rate-limiting per-user dereference operations.

Presenting WebIDs in user interfaces

WebID Profiles may carry human-readable labels (such as foaf:name or rdfs:label) that clients can use for display, but Profiles are not required to include such labels. User interfaces should therefore be designed to display a WebID URI as a fallback identifier and should not assume that a short, human-friendly name is always available for either the agent or any associated storage.
