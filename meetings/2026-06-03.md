## Topics

### Access Control Languages

* Issue: https://github.com/solid/specification/issues/787

- eP: I think I stated it a couple of times, whether it is feasable to implement or not. It seems tricky. I think we need invovlement from implementers of ACP servers. Probably someone from Inrupt needs to chime in.
- CB: I see the abstract concern. It is absolutely possible to have implementation that uses different access control mechanisms jointly. Operational considerations are totally valid. It is not ground to say it is not feasable. Request can have filter to check how it is authorized both under WAC and ACP. System can make decision.
- eP: Enforcing them once set seems fine, but setting and managing them seems complicated. We can spec whatever we want, but we should have implementations proving this.
- JW: I want to echo eP here, that we need to have interest of implementors
- CB: My point is not that I want to write it in spec now. I don't read the issue as writing it in the spec now. What I read and my past experience, this is a proposal that people can give feedback on. I'm not in favor of writing it in the spec and dumping it in WG and implementers. Mostly to signal how it could be done. I wouldn't want to pull the brakes here.
- eP: In the current ecosystem, we have limited implementations. We have a small pool of implementers to check with. 
- CB: My reading of the proposal was less that it should be written into spec today or tomorrow but rather as a proposal for people to consider, provide feedback, and if interested play around with and implement. And if sufficient adoption is present in the community, then this discussion and implementers' feedback could be the basis for a PR to the spec.
