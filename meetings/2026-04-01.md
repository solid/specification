### Web Access Control - Authorization condition with capability detection
https://lists.w3.org/Archives/Public/public-solid/2026Mar/0031.html
https://github.com/solid/web-access-control-spec/pull/134

* eP: Will *creator* related access conditions fit here?
* SC: When we released WAC, we anticipated that there were features that were not included due to lack of engagement. So we included an affordance for extending WAC.
* ... This PR is based on https://github.com/solid/web-access-control-spec/pull/133 by CB addressing client identification. We reached rough consensus which resulted in this PR 134.
* ... This extension of WAC is designed with the following items in mind:
* ... backwards compatability: This update introduces checking conditions but this should not break existing implementations. Therefore, we introduce capability discovery such that a server can advertise whether or not and if so which condition checks are supported such that a client is made aware whether or not a particular condition/rule can actually be enforced by the server.
* ... In the future, if there are more conditions that people are interested, e.g. time based or ODRL, it is still possible to add. Right now, this PR is only defining client and issuer conditions in its core. Other things can be added.
* ... Also, there are corresponding security considerations added, e.g., if a new Server implements "WAC 1.1", and then a server is downgraded to "WAC 1.0", then there might be more rights granted - but this is already explained in the security considerations.
* eP: Big thanks to SC and CB. For me, I currently use ACP because of the client constraint, so when this PR lands, I can also just use WAC.
* ...: There is also the Thread Modelling CG, RZ is also participating there. 
* ...: There might be an argument for requiring a server to throw an error that if a condition cannot be enforced to rule out elevation of access rights. Not going for discussion now, but just wanted to put this out there.

Minutes
* eP: Picking up the discussion from last week - I know that Jeff opened some discussion.
* JW: The problem statement that I see is that we have the default configuration of CSS does not include a pim:storage predicate which then results in clients being required to look up the storage in the storage description. As CS pointed out, you cannot assume that a WebID is stored in the storage that belongs to that WebID.
...: I acknowledge the different use cases for 0 storages and for multiple storages. However, given the most present use case is that a user has a storage and just uses this, the idea was to simply require it. I am not sure whether a spec modification is the right solution or whether it is just the default of CSS.
* JZ: There is no required pointer in the root of the storage. What we should really have is the WebID profile to point to a Solid Server, e.g. using rdfs:seeAlso. This then also might not be publically available.
* eP: Discovery seems to be in limbo. I think Solid26 should point this out. I think it is acceptible to get started. Also, there is the solid:owner. I am under the impression that trying to resolve this in 4 weeks is ambitious.
* SC: I think it might be simplest to get CSS to move a little bit. This might be the simplest fix. As we know Solid, it is about empowering people and controlling their own data. Web Architecture also talks about URI ownership. If an implementation is choosing to not disclose a storage out of the box, that is their implementation choice. I do not know whether that is the easiest way to align with the vision of Solid. At the end of the day, the URI owner is the one to decide whether or not to disclose a storage or my friends. We should give as much room to people that own URIs but from a spec point we cannot require something to be there because as soon as someone says that they do not want to disclose something then this conflicts with Solid's vision. If something is workable enough, then it is good enough :)
* JW: I think I agree with CB and SC. So, then a client should derefence WebID and also dereference rdfs:seeAlso and then look for pim:storage.
* SL: I don't think so. I do not have an opinion on where we put these things. I am not arguing about orthogonality. I am bringing an implementors perspective. 
... I know that the main way is to start at the identity of the user and then discover the storage. If not discoverable, then prompt the user to tell the app what storage to use.
* SC: I think that kind of guideline is fine. My implementation experience is that sometimes you really end up in palce where you cannot assume anything anymore. There is all the error handling that you need to consider. pim:storage is only one way to discover the storage. We are looking at a slice of the problem. The bottom line is that you cannot assume what is there today is there tomorrow. Clients need to be a bit more robust.
A guideline can be along the lines of what SL says, that is good. 
But I do not think that the specs need to be change for that.
* RZ: Shouldnt this be a client-library thing? This could be addressed by a standard library such that a developer does not need to think about all the individual ways. All the error handling would then also come out of the box. 
... Maybe we are solving this problem in the wrong direction?
