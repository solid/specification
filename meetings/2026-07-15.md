### Proposal for Application Capability spec
URL: https://github.com/dokieli/application-capability

Source: https://github.com/dokieli/application-capability

* SC: Some of the features are drawn from prior discussion and implementation experience (from pre 2020 to today, e.g., https://github.com/solid/authorization-panel/issues/55 , https://github.com/solid/specification/pull/253 , https://github.com/solid/authorization-panel/blob/main/proposals/ldn-access-request.md , https://github.com/dokieli/dokieli/commit/edd43f7ef70ed49aee76aff357f11d876019854c , https://github.com/w3c-cg/solid/issues/57#issuecomment-3724332093 ).
* SC: Coauthors/editors: Virginia and I.
* ... it pulls quite a bit of info from prior discussion in the Solid CG.
* ... we are putting it together as a coherent thread.
* ... it is not particularly tied to Solid.
* ... you can technically have these types of capabilities in different kinds of contexts: Solid, OAuth, ... etc.
* ... it is JSON-LD friendly, so it should fit Solid quite fine.
* ... We did not have a spec for all the things that myself and others championed in the CG over the years.
* ... we did not really go out and "incubate"
* ... but that is why we put this document together.
* ... where are capabilities useful?
* ... software discovering capabilities/invocation patterns etc
* ... which are already implemented in dokieli
* ... which others in the CG are currently tackling as well 
* ... e.g. how can you **open** a resource **with** an app 
* eP: How does that compare with the "intent handover" proposal?
* SC: That is a subset of the application capabilities that we cover here
* ... I see that proposal as a very initial proposal
* ... there was also a discussion in the PodOS realm
* ... it was already in 2024 when we wanted to write a spec about it
* ... and now we are here.
* ... There are alot of considerations that may or may not directly be applicable to Solid but there are also those that are applicable.
* ... So, we aim to be meaningful in the Solid Context but also elsewhere.
* ... Capabilities can be invoked.
* ... So, if an application can "open with" certain resources, then the app can advertise that.
* ... We are re-using the same roles as LDN, which will become clear in a second.
* ... Interoperability among classes of products...
* ... The spec's products are organized into modules, capabilities and notifications.
* ... Capabilities basically say "I am able to accept certain mediatypes or certain types of classes"
* ... Some applications might want to express their capabilities in terms of a shape.
* ... We do not restrict that.
* ... We only facilitate the discovery of that.
* ... The senders and receivers decide what to support.
* ... Actions are like ODRL actions,
* ... e.g. "display" for essentially reading..
* ... Invocations, as discussed by others, rely on uri templates and variables.
* ... we borrow from hydra.
* ... On CSP, applications can communicate to a server which CSP level they require to function properly.
* ... You can also embedd other description formats, like OAuth
* ... These are just examples, it is hard to convince others of a particular format.
* eP: Question about CSP, are those requirements if you want to host the application or for the application accessing data?
* SC: I think it is the target place that I am trying to access that should make the requested headers available to me.
* ... Think of the client identifier, if that is the request to authorize the client, then CSP is just one additional thing that was requested.
* eP: I recall discussion around hosting an app and then there is CSP that prevents the app to function properly.
* SC: If a server is able to authorize clients, then the server can give the resource with the desired CSP headers.
* SC: Continuing on Notifications.
* ... borrowing from ActivityStreams
* ... so, there is an example notification that says that this app has these needs.
* ... examples, requirements review, access request, you can use ODRL as well
* ... there is also an error response that includes an inbox so clients know where to send the access request for example
* ... Different applications will have different capabilities, will have different invocations they can execute
* ... There are also extensions, that were originally in the main part as Dokieli already implemented it, but maybe not something for every app. Used to get the idea across.
* ... State is interesting, e.g., open the app in a certain state. But that needs a lot more thought and work. 
* ... Both security and workability.
* ... We included a threat model for each feature in the spec.
* ... Threats around that and mitigations.
* ... If an application does not understand a particular invocation or does reject something, that is the applications call. An app should not just execute it, it should process the input first.
* ... And yes, we talk about the fragment, which was implemented in dokieli for a long time (and have shared examples in the CG).u
* ... I want to emphasise that the hash-form is one approach (compared to the query-form), but they are not particular to invocations - that is not meant for invocations, it is actually in the realm of the resource. That is not really something in the applications domain.
* ... You can also not expect that whatever you put into the query is meaningful (sub-) description of the resource.
* ... We cover the fragment-query aspect in a section. It is a convention.
* ... There were other patterns.
* ... Our PROPOSAL is to continue work on this spec for application capabilities and invocations.
* CB: Would you like to propose your work formally to the group?
* SC: I can follow up with an issue for a work item.
* CB: That would be great, and we can go from there.
